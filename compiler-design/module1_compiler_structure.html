Topic 1

Compiler Overview

Page 3 of 10

Definition

A compiler is a special program that translates source code written in a high-level language (HLL) into an equivalent program in a target language (usually machine code or assembly), while reporting errors found during translation.

Role of Compilers

  • Bridge between HLL and hardware (machine code)
  • Enable programmer productivity using readable syntax
  • Perform error detection (syntax, semantic errors)
  • Optimize generated code for speed/memory
  • Allow platform-independent programming
  • Enable modular software development

HLL → Machine Language

// High-Level Language (C) int a = 5 + 3 * 2; // Assembly (x86) MOV EBX, 3 IMUL EBX, 2 ADD EBX, 5 MOV [a], EBX // Machine Code (Binary) 10111011 00000011 ...

⚠️ Note: Java uses both — javac compiles to bytecode; JVM interprets bytecode. This is called a hybrid approach.

Types of Compilers

Single-Pass
Processes source once. Fast but limited optimization.
Multi-Pass
Multiple scans. Better error handling and optimization.
Cross Compiler
Compiles for a different target platform.
Just-In-Time (JIT)
Compiles at runtime. Used in Java/.NET.

Analysis–Synthesis Model of Compilation

Page 4 of 10
📖 Overview: Compilation is divided into two major conceptual halves — Analysis (understanding the source) and Synthesis (constructing the target). Together they represent the complete compilation process.

🔍 Analysis Phase Front End

  • Reads and understands source code
  • Breaks it into constituent pieces
  • Imposes grammatical structure
  • Detects errors (lexical, syntax, semantic)
  • Collects info into symbol table
  • Creates Intermediate Representation (IR)
Phases: Lexical → Syntax → Semantic
→

🔨 Synthesis Phase Back End

  • Constructs the target program
  • Uses IR produced by analysis
  • Optimizes intermediate code
  • Generates target machine code
  • Manages registers and memory
  • Produces executable output
Phases: ICG → Optimization → Code Gen

Topic 3 · Part A

Phases of a Compiler (1–3)

Page 5 of 10
🎯 Exam Focus: The phases are the #1 most tested topic. Know Input → Output → Responsibilities → Example for each phase.
Phase 1 — Lexical Analysis ★
Also called: Scanner / Tokenizer
Input: Stream of characters (source code)
Output: Stream of tokens
Uses: Regular Expressions, Finite Automata (DFA/NFA)

Responsibilities:

  • Read source character by character
  • Group characters into meaningful lexemes
  • Convert lexemes into tokens: <token-type, attribute>
  • Remove whitespace, comments, blank lines
  • Report invalid token errors
  • Interact with symbol table
Example Transformation:
Input: position = initial + rate * 60 Tokens output: <id,1> <=> <id,2> <+> <id,3> <*> <60> Token types: id → identifier +,-,*,= → operators 60 → integer_literal
Lexeme vs Token: "rate" is the lexeme; <id,3> is the token.
Phase 2 — Syntax Analysis ★
Also called: Parser
Input: Stream of tokens
Output: Parse Tree / Abstract Syntax Tree (AST)
Uses: Context-Free Grammar (CFG), Parsing algorithms

Responsibilities:

  • Check token sequence against grammar rules
  • Detect syntax errors (missing semicolons, brackets)
  • Construct parse tree / AST
  • Types: Top-Down (LL) and Bottom-Up (LR) parsing
Example Transformation:
Input Tokens: <id,1> <=> <id,2> <+> <id,3> <*> <60> AST structure: assign / \ id(1) + / \ id(2) * / \ id(3) 60
Phase 3 — Semantic Analysis ★
Also called: Type Checker
Input: AST + Symbol Table
Output: Annotated AST / IR
Uses: Attribute Grammar, Type system rules

Responsibilities:

  • Type checking — ensure types are compatible
  • Scope resolution — variables declared before use
  • Array bound checking (if static)
  • Function signature matching
  • Type coercion (implicit type conversion)
Example Transformation:
Input: position = initial + rate*60 Checks performed: - id,1 (float) = id,2 (float) + id,3 (float) * 60 (int) Type coercion added: position = initial + rate * inttofloat(60) Annotated AST: each node has type info
Error example: Assigning a string to an int variable is caught here.

Topic 3 · Part B

Phases of a Compiler (4–6)

Page 6 of 10
Phase 4 — Intermediate Code Generation ★
Also called: ICG / IR Generator
Input: Annotated AST
Output: Three-Address Code (TAC), Quadruples, or Bytecode

Responsibilities:

  • Translate AST to machine-independent IR
  • Each instruction has at most 3 operands
  • Use temporary variables (t1, t2 …)
  • Platform-independent representation
  • Simplifies the code generation step
Forms of IR: Quadruples (op, arg1, arg2, result) · Triples (no result field) · SSA (Static Single Assignment)
Input: position = initial + rate*60 After semantic: rate * inttofloat(60) Three-Address Code output: t1 = inttofloat(60) t2 = id[3] * t1 // rate*60.0 t3 = id[2] + t2 // initial+t2 id[1] = t3 // position=t3
Phase 5 — Code Optimization
Also called: Optimizer
Input: Three-Address Code / IR
Output: Optimized IR (same meaning, better performance)

Key Optimization Techniques:

  • Constant Folding: Evaluate constants at compile time (3*5 → 15)
  • Dead Code Elimination: Remove unreachable code
  • Loop Unrolling: Reduce loop overhead
  • Common Sub-expression Elimination: Avoid recomputation
  • Strength Reduction: Replace expensive ops (x*2 → x+x)
  • Function Inlining: Replace function calls with body
Before Optimization: t1 = inttofloat(60) t2 = id[3] * t1 t3 = id[2] + t2 id[1] = t3 After Optimization: // inttofloat(60) → constant 60.0 t1 = id[3] * 60.0 id[1] = id[2] + t1 // Eliminated t2, t3 temporaries
Goal: Preserve program semantics while improving time/space efficiency.
Phase 6 — Code Generation ★
Also called: Code Generator / Back End
Input: Optimized IR
Output: Target assembly / machine code

Responsibilities:

  • Instruction selection — map IR ops to target instructions
  • Register allocation — assign variables to CPU registers
  • Instruction ordering — arrange for pipeline efficiency
  • Handle memory allocation and addressing modes
  • This phase is machine-dependent
Input (Optimized IR): t1 = id[3] * 60.0 id[1] = id[2] + t1 Output (x86 Assembly): LDF R2, id[3] // R2 = rate MULF R2, R2, #60.0 // R2 = rate*60.0 LDF R1, id[2] // R1 = initial ADDF R1, R1, R2 // R1 = initial+t1 STF id[1], R1 // position = R1

📋 SYMBOL TABLE
Shared data structure — accessed by ALL phases throughout compilation
⚠️ ERROR HANDLER
Reports errors at every phase — from lexical to code generation

Symbol Table

A data structure that stores information about identifiers (variables, functions, classes) used in the source program.

Stores:

  • Name of identifier
  • Type (int, float, char…)
  • Scope (local, global)
  • Memory location / offset
  • Number of parameters (for functions)
  • Return type (for functions)
💡 Example Entry:
rate | float | 0x1A4 | global

Front End

  • Performs analysis of source language
  • Machine-independent code
  • Produces IR + symbol table
  • Can be reused for different target machines
  • Contains: Lexer, Parser, Semantic Analyzer

Error Handling

PhaseError TypeExample
LexicalInvalid token@variable
SyntaxGrammar violationif x > 5 (missing parens)
SemanticType mismatchint x = "hello"
Code GenResource errorsToo many vars for registers

Middle End

  • Performs machine-independent optimizations
  • Works on the IR
  • Includes loop transformations, inlining
  • Not present in all compilers (single-pass may skip)

Back End

  • Converts IR to target machine code
  • Machine-specific and platform-dependent
  • Handles register allocation, instruction scheduling
  • Different back ends for x86, ARM, MIPS, etc.
  • Contains: Code Generator, Instruction Selector
📌 Key advantage: Front end + Middle end can be reused. Only the back end changes when targeting a new processor architecture.

Tool-Based Approach to Compiler Construction

Page 8 of 10
📖 Overview: Instead of writing a compiler from scratch, compiler-construction tools automate the creation of lexical analyzers, parsers, and code generators from formal specifications.

What are Compiler-Generating Tools?

These are software tools that take a formal specification (grammar rules, regex patterns) as input and automatically generate the corresponding compiler component as output.

Role of Automation in Compiler Design

  • Reduce development time significantly
  • Minimize manual coding errors
  • Enable changes by modifying only the grammar/spec
  • Allow non-experts to build correct parsers
  • Enforce systematic, well-tested techniques
  • Support rapid prototyping of new languages

Compiler Construction Tools ★ Exam Hot

LEX / FLEX
Lexical Analyzer Generator
Takes regex patterns as input. Generates C code for a lexical analyzer (scanner/tokenizer).
YACC / BISON
Yet Another Compiler Compiler
Takes BNF grammar rules as input. Generates LALR(1) bottom-up parser in C.
ANTLR
ANother Tool for Language Recognition
Generates parsers, lexers, tree walkers from EBNF grammar. Outputs Java, Python, C#, and more.
LLVM
Low Level Virtual Machine
Modern compiler infrastructure. Reusable middle and back end. Used by Clang/Swift/Rust.
GCC
GNU Compiler Collection
Open source compiler framework supporting C, C++, Fortran, Ada, and more languages.
SDT Schemes
Syntax-Directed Translation
Attach semantic actions to grammar rules. Automate semantic analysis and IR generation.
💡 LEX + YACC Workflow:
LEX spec (.l) → lex → lex.yy.c
YACC grammar (.y) → yacc → y.tab.c
Compile both with GCC → executable compiler/parser

Important Definitions ★ Exam Hot

Token
A pair <token-type, attribute-value> representing a meaningful unit in the source (e.g., identifier, keyword, operator).
Lexeme
The actual character sequence in the source code that matches a token pattern (e.g., "position", "60", "+").
Pattern
A rule describing the set of strings that can form a lexeme for a token. Written as a regular expression.
Parse Tree
A hierarchical tree representing the syntactic structure of source code according to the grammar rules.
Abstract Syntax Tree (AST)
A condensed parse tree that omits unnecessary nodes (punctuation, trivial grammar rules).
Symbol Table
A data structure storing information about identifiers: name, type, scope, memory location.
Intermediate Representation (IR)
A machine-independent form of code between source and target, often Three-Address Code (TAC).
Three-Address Code (TAC)
IR where each instruction has at most one operator and three operands: x = y op z.
Context-Free Grammar (CFG)
A grammar formalism used to specify the syntactic structure of programming languages.
Type Checking
Verification that operands of an operator are of compatible types; done in semantic analysis.
Register Allocation
Assigning program variables to CPU registers during code generation to minimize memory accesses.
Linker
Combines multiple object files and resolves external references to create a single executable.

Important Points to Remember

  • A compiler processes the entire source before outputting.
  • Lexical analysis uses Regular Expressions + DFA/NFA.
  • Syntax analysis uses CFG + Parsing algorithms.
  • Semantic analysis uses Attribute Grammar.
  • ICG produces Three-Address Code (most common form).
  • Optimization does NOT change program meaning, only efficiency.
  • Code generation is the only machine-dependent phase.
  • The Symbol Table is used by ALL phases of the compiler.
  • Front end is source-language dependent; back end is target-machine dependent.
  • LEX generates lexer; YACC generates parser (both produce C code).
  • ANTLR supports multiple output languages (Java, Python, C#…).
  • inttofloat() in TAC represents implicit type coercion.

Typical Exam Questions

  • Define compiler. Explain the role of a compiler. (2 marks)
  • Differentiate compiler and interpreter. (4–6 marks)
  • Explain the analysis-synthesis model with a diagram. (6 marks)

2-Mark Short Answers ★

Q: What is a compiler?
A program that translates source code written in a HLL into target language (machine/assembly code), while reporting errors.
Q: What is a token?
A pair <token-type, attribute-value> — the output of lexical analysis. E.g., <id, 1> for variable "position".
Q: What is a parse tree?
A tree that shows the syntactic structure of a program according to grammar rules. Root is the start symbol.
Q: What is the role of semantic analysis?
Checks type compatibility, scope of variables, and logical correctness. Detects errors like type mismatch.
Q: Define intermediate code.
A machine-independent representation (like Three-Address Code) produced between analysis and code generation phases.
Q: What is LEX?
A compiler construction tool that generates lexical analyzers (scanners) from regular expression specifications.
Q: What is the symbol table used for?
Stores identifier info (name, type, scope, memory location) used by all phases of the compiler.
Q: What is code optimization?
A phase that improves the IR code for better time/space efficiency without changing the program's meaning.
Q: What is the difference between lexeme and token?
Lexeme is the actual string in source ("rate"); token is the abstract representation <id, 3>.

Summary Page
🎯 Last-Minute Revision: This page summarizes everything you need to remember. Read this just before your exam!
📘 Compiler Basics
  • HLL → Machine code translator
  • Compiles entire program at once
  • Produces object code on disk
  • Faster execution than interpreter
  • Examples: GCC, g++, javac
🔍 Analysis Phase
  • = Front End
  • Machine-independent
  • Phases: Lex → Syn → Sem
  • Output: IR + Symbol Table
  • Detects all language errors
🔨 Synthesis Phase
  • = Back End
  • Machine-dependent
  • Phases: ICG → Opt → CG
  • Input: IR; Output: Machine code
  • Different per target platform
🔤 Lexical Analysis
  • Characters → Tokens
  • Uses RegEx + DFA
  • Removes whitespace/comments
  • Tool: LEX / FLEX
🌳 Syntax Analysis
  • Tokens → Parse Tree / AST
  • Uses CFG + parsing algos
  • LL(top-down) or LR(bottom-up)
  • Tool: YACC / BISON / ANTLR
✅ Semantic Analysis
  • AST → Annotated AST
  • Type checking, scope check
  • Type coercion (inttofloat)
  • Uses Attribute Grammar
📝 ICG
  • AST → TAC (IR)
  • Three-address code: x=y op z
  • Uses temp vars (t1, t2...)
  • Machine-independent IR
⚡ Optimization
  • IR → Better IR (same meaning)
  • Constant folding, dead code
  • Loop unrolling, inlining
  • Preserves program semantics
💻 Code Generation
  • IR → Assembly/Machine code
  • Register allocation
  • Instruction selection
  • Machine-dependent phase

Complete Phase Flow — One-Line Summary

Source → Lexer→Tokens → Parser→AST → Semantic→IR → ICG→TAC → Opt→Better IR → CodeGen→ASM
🔑 Key formula to remember:
Compiler = Analysis (FE) + Synthesis (BE)
Analysis = Lex + Syn + Sem
Synthesis = ICG + Opt + CodeGen
