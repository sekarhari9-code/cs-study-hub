Topic 1

Introduction to Syntax Analysis

Page 3

Definition

Syntax Analyzer (Parser): The second phase of a compiler that takes a stream of tokens from the lexical analyzer and builds a parse tree (or abstract syntax tree) representing the grammatical structure of the program according to the language's grammar rules.

Role of Parser in Compiler

The parser performs the following critical functions:

  • Structure Verification: Checks if token sequence follows grammar rules
  • Parse Tree Construction: Builds hierarchical representation of program structure
  • Error Detection: Identifies syntax errors and reports them with meaningful messages
  • Error Recovery: Attempts to continue parsing after detecting errors
  • Symbol Table Updates: Communicates with symbol table for identifier management

Compiler Phase Interaction

Source Code
↓
Lexical Analyzer
↓
Token Stream
↓
Parser (Syntax Analyzer)
↓
Parse Tree / AST
↓
Semantic Analyzer

Parser-Lexer Relationship

Lexical Analyzer

  • Reads characters
  • Produces tokens
  • Deals with lexical structure
  • Uses regular expressions
  • Example: <ID, "x">

Syntax Analyzer

  • Reads tokens
  • Produces parse tree
  • Deals with grammatical structure
  • Uses context-free grammars
  • Example: E → E + T

Importance of Syntax Checking

Why is syntax analysis critical?

  1. Correctness: Ensures program follows language rules before execution
  2. Structure: Reveals program structure needed for semantic analysis and code generation
  3. Early Error Detection: Catches structural errors before runtime
  4. Meaningful Messages: Provides clear error messages to programmers
  5. Foundation: Parse tree is essential for all subsequent compiler phases

Example: Token Stream to Parse Tree

// Source Code x = y + z * 2 // Token Stream from Lexer <ID,x> <ASSIGN> <ID,y> <PLUS> <ID,z> <MULT> <NUM,2> // Parser builds parse tree showing operator precedence

Types of Parsers

Type Direction Technique
Top-Down Root → Leaves Recursive Descent, LL(1)
Bottom-Up Leaves → Root LR, SLR, LALR, CLR
⚠️ Key Point: Syntax analysis uses Context-Free Grammars (CFG) because they are powerful enough to describe most programming language constructs, yet simple enough to parse efficiently.

Topic 2

Context-Free Grammars (CFG)

Page 4

Definition

A Context-Free Grammar (CFG) is a formal notation for specifying the syntax of a programming language. It consists of a set of recursive rules that describe how strings in the language can be formed.

Components of CFG

A grammar G is defined as: G = (V, T, P, S)

Symbol Component Description Example
V Non-terminals Variables that can be replaced E, T, F
T Terminals Actual symbols in the language +, *, id, (
P Productions Rewrite rules E → E + T
S Start Symbol Initial non-terminal E

Grammar Example: Arithmetic Expressions

// Grammar for arithmetic expressions E → E + T // Expression with addition E → T // Expression is a term T → T * F // Term with multiplication T → F // Term is a factor F → (E) // Factor is parenthesized expression F → id // Factor is an identifier

Non-terminals (V)

  • E — Expression
  • T — Term
  • F — Factor

These are syntactic variables that represent sets of strings.

Terminals (T)

  • + — Plus operator
  • * — Multiply operator
  • (, ) — Parentheses
  • id — Identifier

These are the actual tokens from lexical analysis.

Production Rules

Production Format: A → α

  • A is a single non-terminal (left side)
  • α is a string of terminals and/or non-terminals (right side)
  • → means "can be rewritten as" or "produces"
  • Multiple alternatives can be written with |: A → α | β

Example: Parsing "id + id * id"

Step-by-step derivation:

E ⇒ E + T // Apply E → E + T ⇒ T + T // Apply E → T ⇒ F + T // Apply T → F ⇒ id + T // Apply F → id ⇒ id + T * F // Apply T → T * F ⇒ id + F * F // Apply T → F ⇒ id + id * F // Apply F → id ⇒ id + id * id // Apply F → id

This grammar correctly captures that * has higher precedence than +

⚠️ Key Concept: In CFG, any substitution of a non-terminal is allowed regardless of context, hence "context-free". This makes CFGs more powerful than regular expressions but simpler than context-sensitive grammars.

Topic 3

Derivations & Parse Trees

Page 5

What is Derivation?

A derivation is a sequence of production rule applications that transforms the start symbol into a string of terminals (a sentence in the language). Each step replaces one non-terminal with the right-hand side of a production.

Types of Derivations

Leftmost Derivation

Always replace the leftmost non-terminal first.

// String: id + id E ⇒ E + T ⇒ T + T ⇒ F + T ⇒ id + T ⇒ id + F ⇒ id + id

Used in top-down parsing

Rightmost Derivation

Always replace the rightmost non-terminal first.

// String: id + id E ⇒ E + T ⇒ E + F ⇒ E + id ⇒ T + id ⇒ F + id ⇒ id + id

Used in bottom-up parsing

Parse Trees

A parse tree (or concrete syntax tree) is a graphical representation of a derivation that shows the hierarchical structure of the derived string.

  • Root: Start symbol (S or E)
  • Internal Nodes: Non-terminals
  • Leaves: Terminals (read left-to-right gives the input string)
  • Parent-Child: Production rule application

Parse Tree Example: "id + id * id"

                    E
                    |
            --------|--------
            |       |       |
            E       +       T
            |               |
            T         ------|------
            |         |     |     |
            F         T     *     F
            |         |           |
           id         F          id
                      |
                     id

This tree shows that * is applied before + (correct precedence)

Derivation vs Parse Tree

Aspect Derivation Parse Tree
Representation Sequential list of steps Hierarchical tree structure
Order Shows order of rule application Order-independent (structure only)
Uniqueness Multiple derivations possible Unique for unambiguous grammar
Detail Shows each step explicitly Shows final structure compactly

Abstract Syntax Tree (AST)

An AST is a condensed form of parse tree that removes redundant information and focuses on semantic structure.

Parse Tree:
  • Contains all non-terminals
  • Shows all production steps
  • More verbose
Abstract Syntax Tree:
  • Only essential nodes
  • Removes intermediate productions
  • More compact, easier to process
💡 Important: Leftmost and rightmost derivations produce the same parse tree for unambiguous grammars, just in different orders. The parse tree captures the structure, not the derivation order.

Ambiguity & Associativity

Page 6

Ambiguous Grammar

Definition:

A grammar is ambiguous if there exists at least one string that has more than one parse tree (or equivalently, more than one leftmost/rightmost derivation).

Example: Ambiguous Expression Grammar

// Ambiguous grammar E → E + E E → E * E E → id

Problem: String "id + id * id" has TWO parse trees

Parse Tree 1: (id + id) * id 
        E
        |
    ----|----
    |   |   |
    E   *   E
    |       |
  --|--    id
  | | |
  E + E
  |   |
 id  id

Evaluates to: (3 + 5) * 2 = 16

Parse Tree 2: id + (id * id) 
        E
        |
    ----|----
    |   |   |
    E   +   E
    |       |
   id     --|--
          | | |
          E * E
          |   |
         id  id

Evaluates to: 3 + (5 * 2) = 13

⚠️ Problem: Two different parse trees mean two different meanings! This is unacceptable for programming languages where semantics must be unambiguous.

Operator Precedence

Precedence determines which operator is evaluated first in an expression without parentheses.

Operator Precedence Example
Parentheses ( ) Highest (1) (2 + 3) evaluated first
Exponentiation ^ High (2) 2 ^ 3 before multiplication
Multiplication *, Division / Medium (3) 3 * 4 before addition
Addition +, Subtraction - Low (4) Evaluated last

Associativity

Associativity determines evaluation order when operators have the same precedence.

Operator Associativity Expression Evaluation
+, -, *, / Left a - b - c (a - b) - c
^, = Right a = b = c a = (b = c)

Resolving Ambiguity

Strategy: Rewrite grammar to encode precedence and associativity

// Unambiguous grammar (precedence: * > +) E → E + T // + has lower precedence E → T T → T * F // * has higher precedence T → F F → (E) F → id

Key Idea: Higher precedence operators appear lower in the grammar hierarchy.

Dangling Else Problem

// Ambiguous if-else grammar S → if E then S S → if E then S else S // Ambiguous: which if does else match? if E1 then if E2 then S1 else S2

Convention: Match else with the closest unmatched if (most languages)

🎯 Exam Tip: To prove ambiguity, show two different parse trees for the same string. To resolve it, rewrite the grammar with proper precedence levels or add constraints.

Top-Down Parsing

Page 7

Definition

Top-Down Parsing constructs the parse tree starting from the root (start symbol) and proceeding toward the leaves (terminals). It attempts to find a leftmost derivation for the input string.

Concept

Direction: Root → Leaves (Start Symbol → Input String) 
Start Symbol (E)
       ↓
  Expand using
  productions
       ↓
   Intermediate
   derivations
       ↓
  Terminal string
   (input tokens)

Top-Down Parsing Methods

1. Recursive Descent

  • Uses recursive procedures
  • One procedure per non-terminal
  • May require backtracking
  • Simple to implement manually
  • Suitable for small grammars

2. Predictive Parsing

  • Table-driven approach
  • No backtracking required
  • Uses FIRST and FOLLOW sets
  • Works for LL(1) grammars
  • More efficient than RD

Parsing Process Example

Grammar:

E → T E' E' → + T E' | ε T → F T' T' → * F T' | ε F → (E) | id

Input: id + id

1. Start with E, match against input 2. E → T E' // Expand E 3. T → F T' // Expand T 4. F → id // Match id with input 5. T' → ε // No *, use ε 6. E' → + T E' // Match +, expand T 7. T → F T' // Expand T again 8. F → id // Match second id 9. T' → ε // No more operators 10. E' → ε // End of input

Advantages and Disadvantages

Advantages Disadvantages
✓ Easier to understand and implement ✗ Cannot handle left-recursive grammars
✓ Error detection is straightforward ✗ May require backtracking (inefficient)
✓ Produces leftmost derivation naturally ✗ Works only for LL grammars (subset of CFG)
✓ Good error messages possible ✗ Less powerful than bottom-up parsers

Requirements for Top-Down Parsing

For efficient top-down parsing without backtracking:

  1. No Left Recursion: Grammar must not have productions like A → Aα
  2. Left Factored: Common prefixes should be eliminated
  3. LL(1) Property: Should be able to predict production using 1 lookahead token
  4. Deterministic: No ambiguity in choosing productions

Visual Comparison

Top-Down (LL)
• Start: Root
• End: Leaves
• Direction: ↓
• Build: Predictive
• Derivation: Leftmost
Bottom-Up (LR)
• Start: Leaves
• End: Root
• Direction: ↑
• Build: Shift-Reduce
• Derivation: Rightmost (reverse)
⚠️ Key Point: Top-down parsing is ideal for hand-written parsers and when you need clear error messages. However, it requires grammar transformations (left recursion elimination, left factoring) to work efficiently.

Topic 6

Recursive Descent Parsing

Page 8

Definition

Recursive Descent Parsing is a top-down parsing technique that uses a set of recursive procedures to process the input. Each non-terminal in the grammar has a corresponding procedure.

Key Concepts

  • One Procedure per Non-terminal: Each grammar non-terminal has its own function
  • Recursive Calls: Procedures call each other based on production rules
  • Lookahead: Examine next token to decide which production to use
  • Backtracking: If a choice fails, try another production (inefficient)

Example Grammar (Without Left Recursion)

// Grammar suitable for recursive descent E → T E' E' → + T E' | ε T → F T' T' → * F T' | ε F → (E) | id

Recursive Descent Implementation

Procedure for E

procedure E() { T(); // Call procedure for T E_prime(); // Call procedure for E' }

Procedure for E'

procedure E_prime() { if (lookahead == '+') { match('+'); // Consume + token T(); // Parse T E_prime(); // Recursively parse E' } // else: ε production, do nothing }

Procedure for T

procedure T() { F(); T_prime(); }

Procedure for T'

procedure T_prime() { if (lookahead == '*') { match('*'); F(); T_prime(); } }

Procedure for F

procedure F() { if (lookahead == '(') { match('('); E(); // Parse expression inside parentheses match(')'); } else if (lookahead == ID) { match(ID); // Match identifier } else { error("Syntax error"); } }

Helper Functions

// Global variable for current token Token lookahead; procedure match(Token t) { if (lookahead == t) { lookahead = getNextToken(); // Get next token from lexer } else { error("Unexpected token"); } } procedure parse() { lookahead = getNextToken(); // Initialize with first token E(); // Start parsing from start symbol if (lookahead != EOF) { error("Unexpected symbols after end of input"); } }

Backtracking Problem

Issue with ambiguous production choices:

// If grammar has: S → ab | ac // Parser must try first alternative // If it fails, backtrack and try second // This is INEFFICIENT!

Solution: Transform grammar to be predictive (eliminate backtracking using left factoring and FIRST/FOLLOW sets)

⚠️ Limitations of Recursive Descent:
  • Cannot handle left-recursive grammars directly (causes infinite recursion)
  • Backtracking can be very inefficient
  • Difficult to implement for complex grammars
  • Better approach: Predictive parsing with parsing table

Topic 7

Left Recursion & Left Factoring

Page 9

Left Recursion

Definition:

A grammar is left-recursive if it has a non-terminal A such that there exists a derivation A ⇒+ Aα for some string α.

// Direct left recursion A → Aα | β // Indirect left recursion A → Bα B → Aβ

Problem: Causes infinite loop in top-down parsers!

Eliminating Direct Left Recursion

Transformation Rule:

Before:
A → Aα₁ | Aα₂ | ... | Aαₘ A → β₁ | β₂ | ... | βₙ
After:
A → β₁A' | β₂A' | ... | βₙA' A' → α₁A' | α₂A' | ... | αₘA' | ε

Example: Arithmetic Expression Grammar

Original (Left-Recursive):
E → E + T | T T → T * F | F F → (E) | id
After Eliminating Left Recursion:
E → T E' E' → + T E' | ε T → F T' T' → * F T' | ε F → (E) | id

Step-by-Step Example

Original Production: E → E + T | T
  1. Identify: A = E, α = +T, β = T
  2. Create: E → T E'
  3. Create: E' → +T E' | ε
// Result matches: T + T + T + ... + T E derives T + T + T // Same language!

Left Factoring

Definition:

Left factoring is a grammar transformation that eliminates common prefixes from productions, making the grammar suitable for predictive parsing.

// Problem: Common prefix α A → αβ₁ | αβ₂

Issue: Parser cannot decide which production to use until it sees beyond α

Left Factoring Transformation

Before:
A → αβ₁ | αβ₂ | αβ₃ A → γ
After:
A → αA' | γ A' → β₁ | β₂ | β₃

Left Factoring Example

Original:
S → if E then S else S S → if E then S
After Left Factoring:
S → if E then S S' S' → else S | ε

Combined Example

Original Grammar After Transformation
A → Aab | Aac | bd | be 1. Left factor: A → Ab | b where b = a[b|c] and b = b[d|e]
2. Eliminate left recursion
⚠️ Key Points:
  • Always eliminate left recursion BEFORE left factoring
  • Both transformations preserve the language generated by the grammar
  • These are necessary for predictive (LL(1)) parsing
  • Indirect left recursion requires more complex elimination algorithms

Topic 8

Predictive Parsing & LL(1)

Page 10

Predictive Parser Concept

Predictive parsing is a table-driven top-down parsing technique that uses FIRST and FOLLOW sets to determine which production to apply, without backtracking.

  • Non-recursive: Uses explicit stack instead of recursion
  • Table-driven: Uses parsing table to decide actions
  • No backtracking: Each decision is deterministic
  • LL(1): Left-to-right scan, Leftmost derivation, 1 lookahead symbol

FIRST Set

Definition:

FIRST(α) is the set of terminals that can begin strings derived from α.

Rules for computing FIRST(X):
  1. If X is terminal: FIRST(X) = {X}
  2. If X → ε: add ε to FIRST(X)
  3. If X → Y₁Y₂...Yₖ:
    • Add FIRST(Y₁) - {ε} to FIRST(X)
    • If Y₁ can derive ε, add FIRST(Y₂) - {ε}
    • Continue until Yᵢ cannot derive ε
    • If all Yᵢ can derive ε, add ε to FIRST(X)

FIRST Set Example

// Grammar E → T E' E' → + T E' | ε T → F T' T' → * F T' | ε F → (E) | id // Computing FIRST sets FIRST(F) = {(, id} FIRST(T') = {*, ε} FIRST(T) = FIRST(F) = {(, id} FIRST(E') = {+, ε} FIRST(E) = FIRST(T) = {(, id}

FOLLOW Set

Definition:

FOLLOW(A) is the set of terminals that can appear immediately to the right of A in some sentential form.

Rules for computing FOLLOW(A):
  1. Place $ in FOLLOW(S) where S is start symbol
  2. If A → αBβ: add FIRST(β) - {ε} to FOLLOW(B)
  3. If A → αB or A → αBβ where ε ∈ FIRST(β): add FOLLOW(A) to FOLLOW(B)

FOLLOW Set Example

// Using same grammar // Step 1: FOLLOW(E) = {$} (start symbol) // Step 2: From E → T E' FOLLOW(T) contains FIRST(E') - {ε} = {+} Since ε ∈ FIRST(E'), FOLLOW(T) also contains FOLLOW(E) // Final FOLLOW sets FOLLOW(E) = {$, )} FOLLOW(E') = {$, )} FOLLOW(T) = {+, $, )} FOLLOW(T') = {+, $, )} FOLLOW(F) = {*, +, $, )}

Predictive Parsing Table Construction

Algorithm:

  1. For production A → α:
    • For each terminal a in FIRST(α), add A → α to M[A, a]
    • If ε ∈ FIRST(α), for each terminal b in FOLLOW(A), add A → α to M[A, b]
    • If ε ∈ FIRST(α) and $ ∈ FOLLOW(A), add A → α to M[A, $]
  2. Mark all undefined entries as error

Example Parsing Table

Non-terminal id + * ( ) $
E E → T E' E → T E'
E' E' → +T E' E' → ε E' → ε
T T → F T' T → F T'
T' T' → ε T' → *F T' T' → ε T' → ε
F F → id F → (E)

LL(1) Grammar Conditions

A grammar is LL(1) if for each production A → α | β:

  1. FIRST(α) ∩ FIRST(β) = ∅ (no common first terminals)
  2. If ε ∈ FIRST(α), then FIRST(β) ∩ FOLLOW(A) = ∅
  3. If ε ∈ FIRST(β), then FIRST(α) ∩ FOLLOW(A) = ∅

Simplified: No ambiguity in choosing which production to use based on one lookahead symbol.

⚠️ Key Point: Predictive parsing works ONLY for LL(1) grammars. The grammar must be unambiguous, non-left-recursive, and left-factored.

Bottom-Up Parsing

Page 11

Definition

Bottom-Up Parsing constructs the parse tree starting from the leaves (input tokens) and proceeding toward the root (start symbol). It attempts to find a rightmost derivation in reverse.

Key Concepts

Direction: Leaves → Root (Input String → Start Symbol) 
Terminal string
(input tokens)
       ↑
  Reduce using
  productions
       ↑
   Intermediate
   reductions
       ↑
  Start Symbol (E)

Shift-Reduce Parsing

Bottom-up parsers use a stack and perform two main operations:

1. Shift

Push the next input symbol onto the stack.

Stack: ... $ Input: id + id $ → Shift Stack: ... $ id Input: + id $

2. Reduce

Replace handle (right side of production) on stack with left side non-terminal.

Stack: ... $ id Input: + id $ → Reduce by F → id Stack: ... $ F Input: + id $

Handle

A handle is a substring of the sentential form that matches the right side of a production and whose reduction represents one step in the reverse of a rightmost derivation.

Key: Identifying the correct handle is crucial for bottom-up parsing.

Parsing Example: "id + id"

// Grammar E → E + T | T T → id
Step Stack Input Action
1 $ id + id $ Shift
2 $ id + id $ Reduce T → id
3 $ T + id $ Reduce E → T
4 $ E + id $ Shift
5 $ E + id $ Shift
6 $ E + id $ Reduce T → id
7 $ E + T $ Reduce E → E + T
8 $ E $ Accept

Other Actions

3. Accept

Parsing completes successfully when stack contains start symbol and input is empty.

Stack: $ E Input: $ Action: Accept

4. Error

No shift or reduce action is possible — syntax error detected.

Stack: $ E + + Input: id $ Action: Error

Advantages of Bottom-Up Parsing

Advantage Description
More Powerful Can parse a larger class of grammars (LR vs LL)
Handles Left Recursion No grammar transformation needed
Earlier Error Detection Detects errors as soon as possible
Widely Used Most parser generators (YACC, Bison) use LR parsing
⚠️ Key Difference: Top-down starts with grammar rules and tries to match input; bottom-up starts with input and tries to build grammar structure. Bottom-up is more powerful but harder to implement by hand.

Topic 10

LR Parsing Techniques

Page 12

LR Parsing Overview

LR Parsing stands for Left-to-right scan, Rightmost derivation (in reverse). It is the most powerful shift-reduce parsing technique.

  • L: Scans input from left to right
  • R: Constructs rightmost derivation in reverse
  • (k): Uses k lookahead symbols (usually k=1)

Types of LR Parsers

Parser Type Full Name Power Table Size
SLR Simple LR Least powerful Smallest
LALR Look-Ahead LR Medium Medium (same as SLR)
CLR Canonical LR Most powerful Largest

Detailed Comparison

Feature SLR LALR CLR
Power Least powerful, simplest More powerful than SLR Most powerful LR technique
States Fewer states Same as SLR Many more states
Lookahead Uses FOLLOW sets Uses lookahead symbols More precise lookahead
Conflicts More shift-reduce conflicts Fewer conflicts Fewest conflicts
Usage Easy to construct Used in YACC/Bison Rarely used (too large)

LR Parsing Table Structure

An LR parser uses two tables:

ACTION Table

  • shift s: Push state s
  • reduce r: Apply production r
  • accept: Parsing complete
  • error: Syntax error

GOTO Table

  • After reduction, determines next state
  • Indexed by [state, non-terminal]
  • Guides parser after reduce action

Example LR Parsing Table

State ACTION GOTO
id + * $ E T
0 s3 1 2
1 s4 acc
2 r2 s5 r2
3 r4 r4 r4

Legend: s3 = shift and push state 3, r2 = reduce by production 2, acc = accept

LR Parsing Algorithm

Stack: [s₀] // Initial state Input: input_string$ while (true) { state = top_of_stack; symbol = current_input_symbol; action = ACTION[state, symbol]; if (action == shift s) { push(symbol); push(s); advance_input(); } else if (action == reduce A → β) { pop 2 * |β| symbols; // Pop β and states state = top_of_stack; push(A); push(GOTO[state, A]); } else if (action == accept) { return "Success"; } else { return "Syntax Error"; } }

Conflicts in LR Parsing

Shift-Reduce Conflict

Parser doesn't know whether to shift next symbol or reduce current handle.

Example: Dangling else problem

Reduce-Reduce Conflict

Parser has multiple possible reductions for the same handle.

Usually indicates ambiguous grammar
⚠️ Key Points:
  • LR parsers are more powerful than LL parsers
  • They can parse almost all context-free grammars
  • LALR is most commonly used (YACC, Bison)
  • LR parsing tables are automatically generated by tools
  • Hand-construction is complex but follows systematic rules
